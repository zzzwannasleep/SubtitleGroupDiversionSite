# 易用性与低复杂度硬约束

## 1. 文档目标

这份文档用于定义项目的非功能性硬约束，重点保证：

- 站点对内部成员来说足够简单、顺手、好用
- 设计不会越做越复杂
- 代码不会越写越重
- Docker 部署不会变成一堆难以维护的环境变量和隐藏依赖

这不是“建议”，而是后续设计和开发都应遵守的基线。

## 2. 总原则

项目必须优先满足以下目标：

1. `方便易用`
2. `简单可维护`
3. `低部署复杂度`
4. `默认可运行`

如果一个方案虽然“更先进”但会明显增加：

- 使用门槛
- 开发复杂度
- 部署难度
- 环境变量数量
- 排障成本

那么默认不采用。

## 3. 用户体验硬约束

### 3.1 页面必须好理解

要求：

- 用户打开页面后能快速知道“这里能做什么”
- 页面主操作必须明确
- 不做复杂工作流式交互
- 不做为了“显得高级”而增加的装饰性交互

禁止：

- 关键按钮藏在二级菜单里
- 一个页面出现多个同等级主按钮
- 操作路径过长

### 3.2 常用操作必须短路径

要求：

- 登录后，从首页或列表页进入下载动作，路径要尽量短
- 上传流程必须直观，字段尽量少
- RSS 页面必须一眼看到自己的地址
- 我的账户页必须能直接看到角色、RSS、passkey 状态

推荐目标：

- 浏览资源到下载，主路径不超过 3 次点击
- 上传资源首屏核心字段不超过 8 个

### 3.3 不为了“炫”牺牲易用

禁止：

- 复杂动效
- 大面积花哨渐变
- 影响阅读的装饰背景
- 需要学习成本的奇怪交互

优先级：

- 可读性高于视觉花样
- 信息清晰高于页面“高级感”

## 4. 前端实现硬约束

### 4.1 组件体系必须统一

要求：

- 只保留一套主组件体系
- 不混多套重型组件库
- 统一背景、卡片、按钮、输入框、表格风格

禁止：

- 一部分页面用 `shadcn-vue`
- 一部分页面再用 `Element Plus`
- 一部分页面再手搓另一套视觉风格

### 4.2 布局数量必须受控

要求：

- 全站布局最多保持 3 套
- 页面必须挂在既有布局下

当前允许：

- `AuthLayout`
- `SiteLayout`
- `AdminLayout`

### 4.3 权限逻辑必须集中

要求：

- 路由权限统一在 router guard 中处理
- 组件权限统一在权限工具中处理
- 不在页面里到处写零散的角色判断

### 4.4 MVP 不做过度前端工程化

MVP 不建议：

- 微前端
- 复杂状态机框架
- 过度抽象的 UI DSL
- 多主题系统
- Storybook 作为上线前阻塞项

## 5. 后端实现硬约束

### 5.1 不做微服务

要求：

- 主站后端保持单体 Django 应用
- XBT 独立作为 Tracker 服务即可

禁止：

- 为了“解耦”过早拆多个业务服务
- 把 RSS、下载、用户、公告拆成多个独立容器

### 5.2 不做重复方案并存

MVP 只保留一套主方案：

- 认证：`Session`
- 主站后端：`Django + DRF`
- Tracker：`XBT`
- 数据库：`MySQL 8`

禁止：

- Session 和 JWT 同时都做成主路径
- 两套权限模型并存
- 两套下载逻辑并存

### 5.3 业务逻辑要清楚，不要过度抽象

要求：

- 有重复和复杂流程时再抽 Service
- 简单查询接口不必过度分层
- 公共工具收敛到 `common`

禁止：

- 每个小逻辑都拆成很多层
- 为了“架构漂亮”引入无必要抽象

## 6. 部署与运维硬约束

### 6.1 Docker 部署必须简单

要求：

- 目标是 `docker compose up -d` 可以拉起完整系统
- 默认部署结构固定
- 服务数量保持最小
- 生产环境优先使用已构建好的镜像，而不是服务器本地临时构建

当前目标容器：

- `frontend`
- `backend`
- `mysql`
- `redis`
- `xbt`
- `nginx`

推荐默认流程：

1. GitHub Actions 构建镜像
2. 推送到 `ghcr.io`
3. 服务器执行 `docker compose pull`
4. 服务器执行 `docker compose up -d`

这样做比“服务器本地 build”更符合低复杂度目标。

### 6.2 环境变量必须最小化

要求：

- 必填环境变量尽量少
- 除密钥和密码外，其他配置优先给默认值
- 本地开发应尽量在复制 `.env.example` 后即可启动

硬约束建议：

- MVP 必填环境变量目标控制在 `10` 个以内
- 新增环境变量必须说明用途
- 如果一个变量可以通过默认值、Compose 服务名、数据库约定解决，就不要新增变量

推荐优先保留为必填的变量类型：

- Django Secret
- MySQL 用户名/密码/库名
- Redis URL
- 站点基础 URL

尽量不要成为必填的变量类型：

- 大量 UI 开关
- 各种 feature flag
- 能从服务发现推导出的 host/port
- 只在极少场景使用的可选项

### 6.3 日志必须默认可用

要求：

- 每个容器都必须输出标准日志
- 默认使用 `docker logs` 可查看
- 不能依赖进入容器翻文件才能排障

### 6.4 部署文档必须傻瓜化

要求：

- 部署步骤尽量短
- 初始化流程尽量明确
- 不依赖大量隐式前置操作

推荐目标：

1. 复制 `.env.example` 为 `.env`
2. 修改少量必要变量
3. 执行 `docker compose pull`
4. 执行 `docker compose up -d`
5. 初始化数据库
6. 创建管理员

## 7. 配置策略硬约束

### 7.1 默认值优先

要求：

- 能写默认值的地方就写默认值
- 能通过服务名推导的地方就不再配置
- 能通过数据库记录管理的业务配置，不放环境变量

### 7.2 配置层级要简单

建议只保留：

- 环境变量
- Django settings
- 数据库中的业务配置

禁止：

- 再额外堆很多 YAML/JSON/TOML 配置文件
- 一部分配置写环境变量，一部分写多个脚本，一部分写代码常量但没人知道

## 8. 依赖控制硬约束

### 8.1 引入依赖必须有明确价值

每新增一个依赖，都应该能回答：

- 它解决了什么问题
- 为什么不能用现有方案
- 它是否会增加部署或维护负担

### 8.2 不引入大而全方案

MVP 不建议引入：

- 复杂工作流引擎
- 消息中间件集群
- 重型搜索引擎
- 复杂配置中心
- 复杂监控栈作为首发阻塞项

## 9. 设计评审与代码评审准则

后续在设计和开发评审时，都要问这几个问题：

1. 这个方案是否让用户更容易用？
2. 这个方案是否让开发更简单？
3. 这个方案是否让 Docker 部署更简单？
4. 这个方案是否新增了不必要的环境变量？
5. 这个方案是否引入了不必要的依赖或抽象？

如果答案偏向“更复杂”，默认退回重做。

## 10. MVP 红线

以下情况视为偏离方向：

- 页面变得像传统 PT 站一样复杂
- 上传流程需要填过多字段
- 部署需要十几二十个环境变量才能跑
- 同一个功能有多种实现路径并存
- 没有标准日志输出
- 前端混用多套组件风格
- 后端为了架构好看做出过多抽象

## 11. 当前结论

当前项目必须坚持：

- 产品上：简单易用
- 前端上：统一、清楚、少花样
- 后端上：单体、够用、少抽象
- 部署上：Docker 简单拉起
- 配置上：环境变量最少化
- 运维上：默认有日志、默认可排障
